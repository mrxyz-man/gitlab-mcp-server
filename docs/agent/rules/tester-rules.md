# Частные правила роли: Tester

1. Тестовая пирамида:
   - unit как базовый слой,
   - integration для контрактов модулей,
   - e2e только для критичных end-to-end сценариев.
2. Для каждого нового функционального блока покрывать:
   - happy path,
   - invalid input,
   - ошибки внешнего GitLab API,
   - таймауты/сетевые ошибки (если применимо).
3. Тест-кейсы именовать по сценарию, а не по внутренней реализации.
4. Не допускать flaky-тестов; любые нестабильные тесты исправлять до merge.
5. Минимум один негативный сценарий на каждый публичный MCP tool.
