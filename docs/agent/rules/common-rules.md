# Общие правила разработки

## Источники истины

1. Актуальная документация библиотек и инструментов проверяется через MCP Context7 перед внедрением изменений.
2. Внутренние решения проекта (ADR, правила, контракты) имеют приоритет над устными договоренностями.
3. Любое спорное решение должно содержать короткое обоснование в задаче/PR.

## Engineering best practices

1. Small changes: делать минимально необходимые изменения без лишнего объема.
2. Fail fast: валидировать входы и рано завершать ошибочные сценарии.
3. Explicit over implicit: явно фиксировать контракты, типы, границы модулей.
4. Reproducibility: команды сборки/тестов должны быть повторяемыми локально и в CI.

## Clean Code (база)

1. Имена должны объяснять намерение (`intent-revealing names`).
2. Функции короткие, с одной ответственностью.
3. Избегать дублирования; выносить общую логику в переиспользуемые модули.
4. Комментарии только там, где невозможно выразить смысл кодом.
5. Ошибки обрабатывать явно, без скрытого подавления исключений.

## Качество

1. Каждый merge должен сопровождаться тестами на измененное поведение.
2. Любая новая публичная точка интеграции должна иметь минимальную документацию.
3. Нельзя мерджить изменения с failing проверками (lint/test/typecheck/build).
